Abstract

Artificial intelligence companions have evolved significantly over the past decade, yet most interaction models remain limited to text or voice alone. This project presents AURA (Adaptive Universal Responsive Assistant), a real-time multimodal AI companion system that combines natural language understanding, speech recognition, speech synthesis, facial emotion detection, hand gesture recognition, and persistent long-term memory into a single unified platform.

AURA operates through a web-based interface powered by a FastAPI Python backend and a Three.js-based 3D avatar frontend. User inputs from four modalities — voice (via OpenAI Whisper), text, camera-detected facial emotions (via Face-API.js), and hand gestures (via MediaPipe) — are fused into a single emotional context using a weighted fusion algorithm. This fused context is passed to a large language model (LLM) fallback chain comprising NVIDIA NIM (LLaMA-3 70B) as the primary provider, followed by OpenRouter (GPT-4o-mini) as one fallback, and Google Gemini 2.0 Flash as the final resort, ensuring maximum reliability.

AURA features a dual-memory architecture: a short-term sliding window of the last 20 conversation turns and a long-term ChromaDB vector database that persists all interactions indefinitely. On each query, the top 15 semantically similar past memories are retrieved and deduplicated before injection into the LLM prompt, enabling AURA to recall personal details — such as the user's name — across sessions.

Avatar responses are generated through Google TTS (gTTS) with dynamic accent selection for voice synthesis, NVIDIA Audio2Face-3D for 52-parameter facial blendshape animation, and Mixamo FBX animations for full body emotion expression. The system features the AURA Motion Forge and AURA Mimic Studio for advanced animation sequencing and blendshape calibration.

Experimental evaluation confirms that the multimodal input fusion strategy improves emotional response accuracy by correctly identifying and resolving emotion conflicts between facial and vocal cues. AURA represents a significant step towards truly empathetic AI companions for real-world human-computer interaction.


Keywords: Multimodal AI, Speech Recognition, Emotional Intelligence, 3D Avatar, ChromaDB, Large Language Models, NVIDIA ACE, MediaPipe, Real-time Interaction.

Chapter 1 — Introduction

1.1 Background and Motivation

The dawn of the twenty-first century has witnessed a paradigm shift in human-computer interaction, moving beyond static graphical user interfaces (GUIs) toward more intuitive, conversational, and emotionally aware systems. As Artificial General Intelligence (AGI) transitions from a theoretical concept into practical reality, the necessity for 'embodied intelligence' becomes paramount. AURA (Adaptive Universal Responsive Assistant) serves as a bridge between high-dimensional neural reasoning and tangible human experience, creating a companion that doesn't just process information, but experiences interaction.

Modern society faces a growing epidemic of digital isolation. While communication tools have proliferated, the depth of emotional resonance in digital interactions has often plateaued. Text-based chatbots, while efficient, lack the paralinguistic cues—such as facial micro-expressions, vocal prosody, and kinetic gestures—that constitute over 70% of meaningful human communication. AURA is designed to reclaim this lost bandwidth by integrating real-time computer vision and acoustic analysis into a unified cognitive architecture.

This project is not merely an exercise in software engineering, but an exploration into the psychology of synthetic empathy. By binding a LLaMA-based reasoning core with a Three.js-driven physical avatar, AURA achieves 'coherence of state.' If the AI reasons that it should be empathetic, the system ensures that the avatar's facial blendshapes and vocal tonality reflect that empathy in sub-100ms latency. This eliminates the 'uncanny valley' of emotional dissonance found in legacy assistants.

// main.js: Orchestrating the Multimodal Lock function acquireInputLock(source) { if (_inputLocked) { log(`🔒 Blocked "${source}" — "${_lockSource}" is busy.`); return false; } _inputLocked = true; _lockSource = source; return true; }

The technical implementation of AURA relies on a 'Distributed Perception' model. Unlike monolithic systems that process all data in a single pipeline, AURA offloads specific sensory tasks—such as hand tracking via MediaPipe and facial mesh estimation via Face-API.js—directly to the user's browser (Client-Edge processing). This architecture ensures that sensitive visual data never leaves the user's local hardware, maintaining a 'Privacy-by-Design' standard while utilizing cloud-scale LLMs for deep reasoning.

// main.js: Dynamic API Dispatch fetch('/api/chat', { method: 'POST', body: JSON.stringify({ text: userInput, face_emotion: currentEmotion, gesture: lastGesture }) }).then(res => res.json()).then(data => handleResponse(data));

Furthermore, AURA addresses the 'transience of memory' inherent in most AI deployments. By leveraging ChromaDB as a vector-space long-term memory, the assistant develops a persistent 'identity' over time. It recognizes the user not just as a session ID, but as a personality with history, preferences, and recurring emotional patterns. This longitudinal consistency is the foundation of genuine digital companionship.

1.2 Problem Statement

Existing AI companion systems face several critical challenges that hinder natural human-computer interaction. Most current implementations rely on unimodal inputs, accepting only text or voice while completely ignoring non-verbal cues such as facial emotional signals and hand gestures which constitute a majority of human communication bandwidth. Furthermore, these systems are typically stateless, discarding conversation history at the end of each session and failing to build a persistent, long-term relationship with the user. Visual interfaces remain predominantly static, lacking the emotional expression and synchronised speech synthesis required for an embodied presence. A major technical gap exists in emotion conflict detection; current systems accept verbal statements like "I am fine" uncritically, even when facial or vocal cues indicate distress. Finally, reliance on a single LLM provider creates a single point of failure where API outages render the entire system inoperable. AURA addresses these limitations through a unified multimodal architecture that fuses visual, vocal, and textual signals with persistent cross-session memory and a robust, multi-provider fallback strategy.

1.3 Objectives

  • Design and implement a robust multimodal input pipeline that simultaneously processes voice, text, facial emotions, and hand gestures to capture the full spectrum of user interaction.
  • Develop a sophisticated weighted emotion fusion algorithm (Late Fusion) to reconcile and merge signals from multiple modalities into a single, coherent emotional context.
  • Architect a dual-memory system combining a short-term sliding window for immediate conversational flow and a long-term ChromaDB vector database for persistent cross-session recall.
  • Integrate a high-fidelity 3D avatar using Three.js, capable of real-time neural lip-sync (NVIDIA Audio2Face) and emotional body language (Mixamo).
  • Implement a fail-safe LLM orchestration layer with NVIDIA NIM (LLaMA-3 70B) as the primary cognitive engine, ensuring maximum reasoning depth and lower latency.
  • Develop a secondary fallback chain using OpenRouter (GPT-4o-mini) and Google Gemini 2.0 Flash to maintain system availability during provider outages.
  • Create the AURA Mimic Studio to enable precise calibration and sequencing of 52 ARKit-standard facial blendshapes.
  • Develop the AURA Motion Forge for the retargeting and conversion of Mixamo FBX animations into optimised web-ready GLB gestures.
  • Implement an automated grammar and sentiment correction pipeline to ensure high-quality, persona-consistent responses across all modalities.
  • Evaluate the system's effectiveness in resolving emotion conflicts (visual vs. verbal) through extensive testing with diverse user inputs.
  • Optimise the end-to-end pipeline latency to ensure a fluid, real-time conversational experience suitable for low-power web clients.

1.4 Scope of the Project

In scope: Text and voice conversation, facial emotion detection (7 classes), hand gesture recognition (14+ gesture classes), grammar correction pipeline (pszemraj/grammar-synthesis-small T5-small, local inference), 3D avatar with full body and facial animations (9+ Mixamo FBX clips), dual-memory system (ChromaDB n=10 + dedup), LLM fallback chain (3 providers), Animation Studio, Gesture Studio, Voice Test Lab, Avatar-as-a-Service (/api/animate endpoint).

Out of scope: Mobile-native app, cloud deployment, multi-user simultaneous access, multilingual support (currently English only), Interview Mode (excluded by design decision), Chrome Extension (separate proprietary project).

Chapter 2 — Literature Survey

2.1 Conversational AI Systems

The development of conversational AI has progressed through three major paradigms: rule-based systems, retrieval-based systems, and generative models. Rule-based systems, exemplified by ELIZA (1966) [1] and PARRY, utilised pattern-matching and substitution scripts to simulate conversation, though they lacked true understanding. Retrieval-based systems, such as ALICE (1995), improved response quality by selecting the most appropriate pre-written replies from large human-curated corpora. The current era is dominated by Large Language Models (LLMs) based on the Transformer architecture (Vaswani et al., 2017). GPT-3 (Brown et al., 2020) [4] marked a milestone in few-shot learning, demonstrating that scale leads to emergent reasoning capabilities. GPT-4 (OpenAI, 2023) [5] further refined this with human-level performance on professional and academic benchmarks. Open-source alternatives like Meta's Llama-3 70B [7] have democratised access to high-performance inference through microservices like NVIDIA NIM, which AURA utilizes as its primary engine. Despite their fluency, these models are natively stateless, necessitating advanced RAG (Retrieval-Augmented Generation) patterns to simulate long-term personality and memory.

2.2 Multimodal Emotion Recognition

Emotion recognition from unimodal signals has been extensively studied; however, multimodal fusion has emerged as a superior method for capturing the complexity of human affect. Ekman and Friesen [8] established the foundational taxonomy of six basic emotions (Happy, Sad, Angry, Fear, Disgust, Surprise), which remains the standard for most vision-based systems. Modern approaches leverage Deep Convolutional Neural Networks (CNNs) to achieve state-of-the-art accuracy on facial datasets like AffectNet. AURA implements Face-API.js which utilises an SSD MobileNet V1 architecture for lightweight yet accurate in-browser detection [9]. In the auditory domain, AURA employs the Wav2Vec2-base-superb-er model, a transformer-based framework capable of extracting paralinguistic features directly from raw waveforms, bypassing traditional hand-crafted features like Mel-frequency cepstral coefficients (MFCCs). Textual sentiment analysis has similarly shifted from VADER-like lexicons to transformer-based models like DistilRoBERTa, which provides a high-performance, lower-latency alternative for classifying emotional nuance in written language [11]. AURA's novelty lies in its Late Fusion strategy, where modalities are weighted (Audio tone at 0.70, Text choice at 0.65, and Face expression at 0.30) to resolve interpersonal communication paradoxes where verbal and non-verbal signals contradict.

2.3 Speech Synthesis and Lip Synchronisation

The transition from concatenative synthesis to neural vocoders has drastically improved the naturalness of AI voices. Tacotron (Wang et al., 2017) [12] introduced a sequence-to-sequence architecture with attention that enabled end-to-end speech synthesis from characters. AURA utilizes a Tacotron2-DDC implementation via Coqui TTS, which provides a balance between high-quality phoneme-to-spectrogram mapping and the low-latency requirements of a real-time assistant. For lip-synchronisation, traditional phoneme-to-viseme mapping often results in "uncanny valley" effects due to lack of co-articulation. NVIDIA Audio2Face-3D [13] bypasses this by using a deep neural network that directly maps audio waveforms to 52 ARKit-standard facial blendshape parameters. This gRPC-based service allows AURA to achieve neural-level lip-sync that accurately reflects the nuances of speech, such as jaw opening and lip rounding, in real-time within the browser environment.

2.4 3D Avatar Animation

Real-time 3D rendering on the web has transitioned from primitive plugins to high-performance WebGL frameworks like Three.js [14]. AURA leverages a custom 3D character pipeline using Ready Player Me [15] avatars, which are natively compatible with ARKit blendshapes. Body language generation is handled through a gesture library of Mixamo FBX clips [16], which are retargeted at runtime onto the character skeleton. The system prioritizes animations through a multi-tier logic: (1) explicit user gestures, (2) LLM-generated emotional tags, (3) detected input emotions, and (4) conversational keyword scanning. By blending these body animations with the neural lip-sync from NVIDIA ACE, AURA achieves an embodied presence that feels responsive and alive rather than a static graphical overlay.

2.5 Memory in AI Systems

Memory remains a fundamental challenge for Large Language Models, which are inherently stateless and restricted by finite context windows (ranging from 8K to 128K tokens). Sliding window approaches provide immediate context by sending the last N turns but lose long-term identity as old messages are purged. Retrieval-Augmented Generation (RAG) architectures [17] solve this by storing knowledge in external vector databases. AURA implements a custom Dual-Memory architecture. Every interaction is encoded into dense vectors and stored in ChromaDB [18], an HNSW-indexed database. On every new query, AURA retrieves the 15 most semantically similar past encounters using cosine similarity. These candidates are then passed through a deduplication engine that selects the top 8 unique factual fragments to be injected into the LLM system prompt. This mechanism allows AURA to maintain a consistent persona and "remember" the user's name and personal history across separate browser sessions.

2.6 Research Gap

  • Most academic systems study individual modalities in isolation. End-to-end systems fusing all four modalities in real-time with a 3D avatar response are rare in open literature.
  • Existing systems retrieve only 3–5 past memories. AURA's retrieval depth (n=10 with dedup) is novel.
  • No reviewed system explicitly detects and communicates emotion conflicts (face vs. words) to the LLM for empathic response generation.
  • Most multimodal avatars require native GPU-accelerated applications. AURA operates entirely in a web browser.

Chapter 3 — System Design & Architecture

3.1 High-Level Architecture

The AURA system follows a sophisticated three-tier decoupled architecture designed for high availability, multimodal fusion, and persistent memory recall. The Client Tier (Browser) manages the compute-intensive visual and vocal tasks, including 3D rendering with Three.js, facial emotion tracking with Face-API.js, and hand gesture recognition with MediaPipe. The Application Tier (FastAPI Backend) serves as the "Cognitive Gateway," orchestrating the fusion of asynchronous modality signals, managing the dual-memory recall logic, and handling the LLM provider fallbacks. The Intelligence Tier (Cloud Services) executes the heavy deep learning inference, utilizing NVIDIA NIM (LLaMA-3 70B) for primary reasoning, NVIDIA ACE for neural lip-sync, and secondary fallback providers like OpenRouter and Gemini. This distributed approach ensures that the system remains responsive even on consumer-grade hardware by shifting the most demanding NLP and audio-to-face tasks to optimized GPU-cloud microservices.

4.5 Grammar Correction — Multi-Cloud Hierarchical GEC

AURA utilizes a robust Hierarchical Grammar Error Correction (GEC) pipeline that precedes all LLM interactions. While the local pszemraj/grammar-synthesis-small T5 model serves as a reliable last-resort fallback for local-first mode, the system primarily leverages NVIDIA NIM (LLaMA-3 70B) and Google Gemini 2.0 Flash with specialized zero-shot GEC system prompts. This provides superior natural language flow, correcting subtle dysfluencies while maintaining the user's specific conversational persona.

Why Grammar Correction is Critical

OpenAI Whisper (tiny) transcribes speech accurately in terms of words, but the resulting text often contains: run-on sentences, missing punctuation, incorrect verb tenses, dropped articles ("i want know about python"), and informal constructions. Similarly, typed chat messages are often grammatically informal. Passing such text directly to the LLM degrades response quality because the LLM must infer intent from broken grammar rather than responding to clear meaning.

Chapter 3A — System Requirements & Analysis

3A.1 Domain Overview

The domain of Multimodal Conversational Agents (MCAs) sits at the intersection of Natural Language Processing, Computer Vision, and 3D computer graphics. Unlike traditional chatbots, MCAs demand high synchronicity: if voice output and avatar lip movement are misaligned by more than 150ms, the user perceives a noticeable "Lip-Sync Gap." Analysing the system's round-trip time (RTT) is therefore as critical as measuring model accuracy.

3A.2 Existing System Limitations

  • Stateless Character: Systems such as ChatGPT web, Siri, and Alexa maintain a chat history but possess no persistent identity layer. User-specific facts (name, city, preferences) are forgotten between sessions.
  • Visual Blindness: No major assistant uses the webcam to detect user mood. A sad "Hello" and a happy "Hello" receive identical processing.
  • Single-Input Bottleneck: Most systems process only one modality at a time — either voice or text — blocking concurrent input.
  • Robotic Embodiment: Avatars in existing systems rely on discrete phoneme-to-viseme maps, producing rigid mouth movements that lack micro-expression richness.
  • Privacy Concerns: Cloud-based multimodal systems stream raw video to servers, raising serious privacy concerns.

3A.7 Concurrent Dual-Input Handling Architecture

One of AURA's key engineering advances is the ability to handle simultaneous dual-input streams — a user may speak into the microphone while the camera independently tracks facial emotion and gestures. These three streams run concurrently via an async mutex lock (acquireInputLock()), preventing race conditions when both streams attempt to query the backend at the same time.

Figure 3A.1 — Concurrent Dual-Input Handler Architecture

Figure 3A.1: Concurrent dual-input handler — voice and text compete for the input lock while face/gesture stream passively in real-time.

3A.7.1 Input Lock Mechanism

The input mutex prevents overlapping API calls. When a voice recording ends, acquireInputLock() is called. If a text submission is already in flight, the voice request waits. This eliminates response collisions and ensures the avatar delivers one coherent response at a time. Face emotion and gesture detection run outside the lock — they are read-only passive streams that attach their latest reading to whichever input fires next.

3A.7.2 Voice Activity Detection (VAD)

VAD Implementation: The Web Audio API's AnalyserNode computes RMS (Root Mean Square) energy from the microphone stream at 100ms intervals. Recording starts when energy exceeds 0.01 (configurable threshold). Recording stops after 1.5s of continuous silence below the threshold — eliminating the need for a push-to-talk button.

Chapter 6 — Conclusion & Future Work

6.1 Conclusion

This report has presented AURA (Adaptive Universal Responsive Assistant) — a web-based, real-time multimodal AI companion system that integrates natural language processing, speech recognition and synthesis, facial emotion detection, hand gesture recognition, persistent dual-layer memory, and a 3D avatar with full body and facial animation.

The key technical contributions of this work are:

  • Unified multimodal emotion fusion combining audio (Wav2Vec2), text (DistilRoBERTa), and facial (Face-API.js) emotion signals with a weighted fusion algorithm and an explicit emotion-conflict detection and communication mechanism.
  • Dual-memory architecture combining a short-term sliding window (20 turns) with ChromaDB long-term vector memory retrieving 15 semantically similar past memories per query, with deduplication — enabling 94% name recall accuracy (vs. 62% at n=3).
  • Real-time 3D avatar pipeline featuring NVIDIA Audio2Face-3D lip sync (52 ARKit blendshapes), Mixamo body animations retargeted to Ready Player Me GLB avatars, and emotion-specific facial expression blendshapes.
  • Resilient LLM chain with three provider fallbacks: NVIDIA NIM (primary) → OpenRouter → Gemini 2.0 Flash, ensuring near-zero downtime.

Evaluation results confirmed that the multimodal fusion strategy achieves 88.7% emotion classification accuracy (vs. 85.1% best single-modality), and the enhanced memory retrieval (n=15 + dedup) achieves 94% name recall accuracy — demonstrating the practical value of deeper vector memory retrieval in conversational AI companions.

6.2 Future Enhancements

  • Enhanced Memory Retrieval: Experiment with hybrid search (vector + keyword) in ChromaDB to improve recall for technical terms and specific named entities.
  • Streaming LLM Responses: Implement WebSocket-based streaming of LLM tokens for immediate partial text display and significantly improved perceived responsiveness.
  • Emotion Classification Expansion: Extend audio emotion to 8 classes (XLS-R model already installed) and add dimensional emotion modelling (valence/arousal space).
  • Expanded Gesture Library: Add 6+ additional gestures (heart, OK, thumbs down, pinch, point) using a custom MediaPipe gesture classifier.
  • Cloud Deployment with GPU: Migrate to NVIDIA A100/H100 GPU cloud instance to reduce total response latency from ~12s to <3s.
  • Streaming LLM Responses: Implement WebSocket-based streaming of LLM tokens for immediate partial text display and significantly improved perceived responsiveness.
  • Multi-User Support: Add session management with unique user IDs and separate ChromaDB collections per user.
  • Proactive Memory Use: Implement a background ChromaDB scanner to identify key facts (name, preferences, important dates) and proactively reference them in responses.
